# 小红书拟人化增强设计说明

## 目标

在当前已经跑通的 `BitBrowser + CDP + Patchright + 小红书存草稿` 链路上，进一步增强拟人化操作，但控制在一个中等强度范围内：

- 增强真实操作节奏感
- 降低过于机械的页面行为
- 不引入大量无意义的“表演型操作”
- 不在今天做真实存草稿验证

本次只做“小红书拟人化增强方案 2”。

## 当前状态

当前拟人化逻辑主要在：

- [humanizer.py](/D:/auto_go/src/auto_football/infra/publishers/humanizer.py:1)
- [draft_writer.py](/D:/auto_go/src/auto_football/infra/publishers/xiaohongshu/draft_writer.py:1)

现在已经有这些能力：

- 随机 pause
- 鼠标移动点击
- 逐字输入
- 偶发 typo / 删除修正
- 微滚动
- 假 hover / 空白区游走
- 存草稿前 review scroll

这套逻辑已经能提供基础拟人化，但还偏“动作点缀”，不够像一条有阶段感的发布流程。

## 本次设计方向

采用“中等增强”方案，不追求复杂行为编排，而是让现有发布流程在关键节点更像真实人的节奏。

## 范围边界

### 本次要做的

- 增强页面级噪声
- 增强输入级节奏
- 增强流程级节奏
- 增强条件等待与拟人化动作的结合
- 补充对应单元测试

### 本次不做的

- 不增加新的发布业务分支
- 不做“点开草稿箱再回来”这类高风险干扰行为
- 不做多次真实存草稿验证
- 不在今天反复打小红书真实后台

## 设计内容

### 一、页面级增强

在不影响主流程的前提下，增加更自然的页面噪声：

- 切到图文 tab 后增加一次轻微 pause
- 上传前增加一次 anchor 附近 hover / maybe_noise
- 上传完成后增加一次轻量浏览动作，而不是马上跳到标题输入
- 在正文输入前增加一次轻微停顿或小滚动

这类动作的特点是：

- 风险低
- 对结果影响小
- 但能有效打散“点一下 -> 立刻上传 -> 立刻输入”的机械节奏

### 二、输入级增强

当前已经是逐字输入，本次增强重点不在“更慢”，而在“更像分阶段写”：

- 标题输入完成后增加一次短暂停顿
- 正文输入时按段落或字数区间增加停顿
- 正文输入后增加一次回看 pause
- 标签输入前后增加轻量停顿

必要时可以增加一个小能力：

- `pause_after_title`
- `pause_after_body`
- `pause_before_tags`

这样流程更像人类“先写标题，想一下，再写正文，再看一下，再补标签”。

### 三、流程级增强

流程级增强是这次最重要的部分。

建议加入这些节奏节点：

1. 进入页面后
   - 等待页面稳定
   - 轻微噪声

2. 切到图文模式后
   - 等待切换完成
   - 再判断上传控件

3. 上传完成后
   - 不立刻写标题
   - 先 pause，再进入标题

4. 标题写完后
   - pause
   - 再写正文

5. 正文写完后
   - maybe_noise
   - 再处理标签

6. 存草稿前
   - review scroll
   - pause
   - 再点按钮

这不会改变业务逻辑，但会显著增强“阶段感”。

## 条件等待原则

本次增强不应该只靠固定 sleep。

更好的原则是：

- 能判断页面状态时，优先判断状态
- 状态之间再插入小 pause

例如：

- 图文 tab 切换后，先确认图片上传 input 出现
- 上传后，如果页面有明显变化，优先等待变化出现
- 实在没有可靠状态信号，再用保守 pause

这样比无脑拉长 sleep 更稳。

## 测试策略

今天不做真实存草稿验证，所以测试分为两层：

### 1. 单元测试

使用 fake page / fake locator 验证：

- 新增动作有没有被调用
- 动作顺序是否正确
- 上传后是否有 pause / noise / review 行为
- 标签阶段是否有额外节奏节点

### 2. 低风险真实页面验证

只验证页面状态逻辑，不做真实保存：

- 能否进入发布页
- 能否切到图文 tab
- 上传控件判断是否稳定

这层可以使用真实 BitBrowser 页面，但不做提交动作。

### 今天不做

- 不做多次真实上传
- 不做真实草稿保存回归

真实草稿验证留到明天再做。

## 风险控制

### 不要增加高风险多余动作

本次不做：

- 随机点开草稿箱
- 随机切其他页面
- 反复开关 tab
- 无意义多次点击

这些动作看起来更“像人”，但实际更容易把链路打坏。

### 不要把拟人化变成主逻辑

拟人化只是增强层，不应反客为主。

如果某个动作会显著影响稳定性，应优先保住主流程。

## 成功标准

这次增强成功的标准是：

- 小红书存草稿链路的主流程代码仍然清晰
- 页面、输入、流程三层都有增强
- 单元测试可以证明增强动作真实存在
- 今天不做真实草稿保存也能确认逻辑准备到位
- 明天只需要做一次真实草稿验证即可
