# 1.7 数据源层设计

## 背景

当前项目已经具备基础的内容路由、草稿入库和预览能力，但距离发布自动化稳定读取还有一个关键缺口：数据源层还不够稳，导致草稿质量和字段完整度不稳定。

项目当前不追求一步到位做成“高质量内容系统”，而是先完成一个 `1.7 版可发内容`：

- 比纯基础稿更完整
- 不被复杂数据源阻塞
- 能平滑演进到后续“运营可发”和“高质量内容”

## 目标

为内容生成主链路补齐两类最重要的增强信息：

1. `排名 / 积分背景`
2. `近期状态 / 近5场`

这两类信息必须以“结构化优先、兜底清晰、可渐进增强”的方式进入现有流水线。

## 非目标

本阶段不解决以下问题：

- 伤停、预计首发、赛前新闻深抓
- 真人赛事图大规模抓取
- 即梦等图片生成平台接入
- 发布自动化的最终读取与发文编排
- 为所有内容类型建立完整事实槽位系统

## 成功标准

当一条内容草稿入库时，应尽量稳定具备以下增强字段中的大部分：

- `home_rank`
- `away_rank`
- `home_recent_form`
- `away_recent_form`
- `standings_summary`
- `form_summary`

并满足以下运行目标：

- 没有增强信息时，基础内容链路仍可继续生成草稿
- 有增强信息时，文案优先消费结构化字段
- 数据源失败不会阻塞整条内容生产链
- 后续新增网页抓取时，不需要推翻本阶段数据结构

## 总体策略

采用 `结构化优先 + 少量兜底 + 渐进增强` 的策略。

优先级如下：

1. `P0 保底`
   当前已稳定可用的比赛主数据源，保证比赛名、时间、比分、状态、联赛等基础字段存在。
2. `P1 增强`
   为排名和近期状态补结构化数据。
3. `P2 后续扩展`
   在不改变主模型的前提下继续接网页抓取、深度统计和更强的内容上下文。

## 数据源分工

### 1. API-Football

用途：

- 赛事详情
- 联赛 standings
- 球队最近比赛
- 已有赔率和部分伤停能力

定位：

- 仍然是最优先的结构化足球源
- 第一阶段里优先承担 `近期状态` 和 `联赛排名`

### 2. football-data

用途：

- 比赛赛程
- 联赛表
- 球队最近比赛

定位：

- 结构化补强源
- 优先用于补 standings 和 recent matches
- 必须有请求预算与缓存，避免免费额度失控

### 3. OpenFootball

用途：

- 历史比赛结果
- 近5场战绩兜底

定位：

- 近期状态兜底源
- 不承担真实积分榜角色

### 4. ClubElo

用途：

- 强弱背景
- 排名背景增强

定位：

- 不是积分榜替代品
- 当真实 standings 拿不到时，只用于“背景增强”，不直接冒充联赛排名

## 数据模型设计

### 核心结构化字段

继续复用 `MatchInfo`，重点强化以下字段：

- `home_rank`
- `away_rank`
- `home_recent_form`
- `away_recent_form`

### 派生摘要字段

为了降低内容生成复杂度，新增或补充两个派生摘要：

- `standings_summary`
- `form_summary`

这两个字段不作为源数据真相，而是由结构化字段派生得到的可读文本。

### 设计原则

- 结构化字段优先于文本摘要
- 摘要必须可由结构化字段重建
- 后续新增来源时优先填结构化字段，不直接只塞一段 summary

## 模块边界

为避免继续把逻辑堆进 `pipeline.py`，本阶段建议拆成三个职责单元。

### 1. StructuredMatchProvider

职责：

- 统一封装 API-Football 和 football-data 的结构化拉取
- 输出统一的数据格式

负责内容：

- standings
- recent matches
- competition context

### 2. FormAndStandingsService

职责：

- 只负责补 `rank + recent_form`
- 按优先级从结构化源取值
- 处理缺失与兜底

优先级建议：

1. API-Football standings / fixtures
2. football-data standings / matches
3. OpenFootball recent results
4. ClubElo background fallback

### 3. ContextSummaryBuilder

职责：

- 根据结构化字段生成：
  - `standings_summary`
  - `form_summary`

要求：

- 不直接访问外部源
- 只消费结构化字段
- 便于后续替换文案模板而不动数据层

## 与现有代码的关系

### 保持不动或尽量少动

- `cli.py`
- 路由器主结构
- 图片生成主链路
- 发布适配器
- 预览页面基本框架

### 重点改造区域

- `clients.py`
- `knowledge.py`
- `pipeline.py`
- `schemas.py`
- `config.py`

## 数据流

新的增强数据流应为：

1. 拉取基础比赛信息
2. 路由器决定内容机会
3. enrichment 阶段调用 `StructuredMatchProvider`
4. `FormAndStandingsService` 补齐 `rank + recent_form`
5. `ContextSummaryBuilder` 生成摘要
6. 内容生成器消费基础字段 + 增强字段
7. 草稿入库

## 降级策略

这是本阶段最关键的稳定性要求。

### 当 standings 缺失时

- `home_rank / away_rank` 允许为空
- 可回退使用 ClubElo 作为背景，但不标记成联赛排名
- `standings_summary` 降级为“当前暂无稳定联赛排名数据”

### 当 recent form 缺失时

- 优先继续生成内容
- `form_summary` 降级为“近期走势信息有限”

### 当增强源全部失败时

- 必须继续产出基础稿
- 不允许让整条路由任务失败

## 缓存与限速

### football-data

- 必须加请求预算控制
- 实际运行按低于官方免费额度的保守值执行
- 需要缓存 standings 与 recent matches

### API-Football

- 优先复用已有缓存机制
- 尽量批量化使用已有字段，避免重复请求

### OpenFootball / ClubElo

- 更适合做天级缓存

## 第一阶段验收

第一阶段完成后，应满足：

1. 随机抽取若干草稿，能稳定看到排名或近期状态中的至少一种增强信息
2. 数据源失败时，内容生成仍然完成
3. 内容模板开始引用这些增强字段
4. 代码边界比当前更清晰，不继续把增强逻辑全部塞进 pipeline

## 后续优化方向

### 通往运营可发

- 加入素材来源链接
- 加入更可解释的摘要字段
- 为发布自动化提供更完整的草稿状态和任务元数据

### 通往高质量内容

- 加入网页抓取的事实补充层
- 加入交锋、热点、新闻摘要
- 加入更强的图片来源与图文组合能力

## 风险与约束

### 1. 免费源稳定性

- football-data 免费额度有限
- OpenFootball 覆盖联赛有限
- ClubElo 只能补强弱背景

### 2. 数据含义混淆

- 必须避免把 ClubElo 当作真实 standings

### 3. 逻辑再次回流到 pipeline

- 如果不明确拆边界，后续很容易再次变成大杂烩

## 结论

本阶段最合适的子项目是：

**“为 1.7 版内容补齐排名和近期状态的数据源层，并保证主链路稳定产草稿。”**

它既能直接提升内容质量，又不会阻塞你现在最重要的目标：尽快把整条内容生产链路跑稳，为后续发布自动化接库做准备。
