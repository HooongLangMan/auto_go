# 自动选稿选图设计说明

## 目标

为当前内容系统增加一个很小的自动草稿选择能力：

- 从数据库里已经生成好的内容中
- 自动挑出一篇最适合的平台稿件
- 连同已经生成好的图片一起
- 送入目标平台草稿箱

第一阶段不做：

- 重新跑内容流水线
- 自动正式发布
- 跨平台复杂调度
- 多篇队列化发布

只做：

- 自动挑一篇
- 自动送草稿

## 当前状态

系统现在已经具备这些能力：

- 自动选比赛
- 自动生成候选内容
- 自动排序出 top candidate
- 自动生成图片
- 数据库存有平台级内容和图片路径
- 微信和小红书草稿链路都已经可用

但当前还缺少一个正式、可复用的能力：

- 不是手动指定 `match_id`
- 而是让系统自己从现成内容里选出“今天最适合送草稿的一篇”

## 设计原则

### 1. 不重跑生成，只消费现成产物

这个能力不负责：

- 重新抓数据
- 重新写文章
- 重新画图

它只从数据库里消费现成的：

- `GeneratedContent`
- `MatchInfo`
- `images`

也就是说，它是一个**后置选择层**，不是内容生产层。

### 2. 共用骨架，平台分规则

微信和小红书的自动选稿，不应该完全写成两套系统。

但它们也不应该共享完全相同的评分标准。

所以结构上采用：

- 一套共用候选过滤骨架
- 两套平台评分函数

也就是：

- `base filter`
- `score_for_xhs`
- `score_for_wechat`

## 范围边界

### 第一阶段先落地

实现顺序建议：

1. 先支持 `小红书自动存草稿`
2. 再支持 `微信自动存草稿`

也就是说：

- 设计上是共用一套 selector service
- 实现上先接 `xiaohongshu`
- `wechat` 随后补充

### 本次不做

- 不做“多个平台同时投递”
- 不做“一次自动挑多篇”
- 不做正式发布
- 不做复杂权重学习

## 候选来源

候选只能来自数据库里已经存在的内容。

来源建议：

- `db.get_preview_payloads(...)`
  用于快速拿最近比赛的聚合内容
- 或 `db.get_match_bundle(match_id)`
  用于单场读取

选择层不直接碰原始 `contents` 表 SQL 逻辑，优先复用现有 DB 聚合能力。

## 共用过滤规则

不管平台如何，先经过一层基础过滤：

- `status == ready_to_publish`
- 必须存在对应平台内容
- 必须存在 `title`
- 必须存在 `content`
- 必须不是空图（如果平台要求图片）
- 必须是最近比赛或最近生成内容

这层目标不是评分，而是先排除明显不能用的候选。

## 小红书评分规则

小红书优先关注：

- 内容长度适合短内容
- 是否有图片
- 是否有更好的封面图
- 标题是否更像封面标题
- 是否更适合快速入草稿

建议第一版简单规则：

- 有图片：加分
- 内容长度在 `200-500` 区间：加分
- 标题更短、更像观点句：加分
- `mode=pre_match`：默认优先
- `mode=result_flash`：次优先
- 没有图片：降权或直接过滤

## 微信评分规则

微信优先关注：

- 长文长度是否达标
- 内容是否更完整
- 信息密度是否更高
- 图片不是核心，但有图更好

建议第一版简单规则：

- 正文字数达到微信下限：加分
- 内容长度更长：加分
- `mode=pre_match` 或 `result_flash` 视需求加权
- 太短的内容直接过滤

## 图片选择规则

这个能力不重新生成图片。

它只做：

- 优先选 `content.images[0]`
- 如果 `images` 为空，才考虑 `remote_images`
- 如果平台强依赖图片而没有图，直接过滤掉候选

所以这里的“选图”本质上是：

- 从现有候选内容绑定的图片里选主图

不是重新做图像策略。

## 服务结构建议

建议新增一个轻量服务：

- `AutoDraftSelectorService`

职责：

1. 拉取最近候选内容
2. 按平台做基础过滤
3. 调对应平台评分函数
4. 返回一个最佳候选

输出建议包含：

- `match_id`
- `platform`
- `score`
- `reason`
- `content`
- `match`

这样 CLI 层拿到结果后可以直接投草稿。

## CLI 设计

第一阶段可以加两个命令：

- `xhs-auto-draft`
- `wechat-auto-draft`

行为：

- 从最近候选里自动挑一篇
- 直接送草稿
- 打印选择结果和发布结果

必要时可以加一个公共命令：

- `auto-draft --platform xiaohongshu|wechat`

但第一版为了简单，分平台命令更清楚。

## 错误处理

第一版要明确区分这几种失败：

1. `no_candidate`
   没有合格候选
2. `missing_image`
   平台要求图片，但没有现成图片
3. `draft_failed`
   候选选出来了，但平台草稿保存失败

这样你后面看日志时，不会分不清到底是“选不到”，还是“发不进去”。

## 测试策略

先只做非破坏性测试：

### 1. 单元测试

验证：

- 共用过滤规则
- 小红书评分规则
- 微信评分规则
- 在多篇候选中是否能选出预期结果

### 2. CLI 层假发布测试

mock 发布器，只验证：

- 选中哪一篇
- 是否把正确的 `PublishBundle` 交给发布器

### 暂不要求

- 今天不做真实多次草稿投递
- 真实平台验证只保留一枪式手动执行

## 成功标准

第一阶段成功的标准是：

- 系统能从最近候选里自动选出一篇小红书稿
- 能从最近候选里自动选出一篇微信稿
- 选稿逻辑是共用骨架、平台分规则
- 不需要再手动找 `match_id`
- 不重跑内容流水线也能直接进草稿流程
